# 历代时间轴页面优化计划

> 审核视角：游戏设计第一性原理 × 宝可梦 VGC 专家
> 审核日期：2026-04-21
> 审核对象：`docs/index.html` 历代时间轴板块 + `report_data.json` 时间轴数据 + `app.py` Tab 2 机制时间轴

---

## 一、审核框架：三个根本问题

在动手之前，先问三个元问题——任何偏离这三个问题的改进，都是在优化错误的目标。

**Q1：谁是历代时间轴的真正用户？**
不是"查资料的人"，而是**正在设计多人对战系统的游戏设计师**。他们使用时间轴只有一个目的：找到"某个设计问题在历史上有哪些解法"，以及"这些解法产生了什么后果"。这决定了时间轴的核心价值是**因果链**，而不是年表罗列。

**Q2：时间轴与报告其他板块的关系是什么？**
时间轴是整个报告的**导航索引层**。10条原则是结论，47条检查清单是方法论，时间轴是史料——用户应该从时间轴出发，点击某个事件，追溯到相关原则和检查清单。当前的问题是，时间轴与其他板块是**断裂的**。

**Q3：当前时间轴的最大系统性缺陷是什么？**
数据质量参差不齐（有的来自官方日志，有的来自 AI 推断），但**没有任何置信度标注系统**。用户无法判断眼前这条信息是否可靠。这不是 UI 问题，是**数据架构缺陷**。

## 二、信息内容审核

### 2.1 现有数据质量总览

当前 `report_data.json` 中的时间轴数据按来源分为四类：

| 来源 | 类型 | 可信度 | 占比估算 |
|------|------|--------|---------|
| Serebii.net 官方更新日志 | 官方记录 | 高 | ~20% |
| 游戏发售日期/版本号 | 事实性 | 高 | ~10% |
| Smogon 社区数据 | 半官方 | 中 | ~10% |
| AI 分析归纳 | 推断 | 低-中 | ~60% |

**核心问题**：60% 的内容依赖 AI 推断，但这些推断没有经过结构化的置信度标注。

### 2.2 内容逐项审查

#### A. 奠基类事件（foundation）

- **Gen 1（1996）** — Pokemon 红绿版发售：✅ 事实准确，是正确的叙事起点
- **Gen 2（1999）** — 携带道具+培育系统：✅ 准确，但需注意"金银版本身不包含双打对战"这个关键事实被埋在 detail 里，用户可能误以为 Gen 2 就有双打
- **Gen 3（2002）** — 双打+特性系统+天气革命：✅ 三件事混在一起，作为单一事件不合理——双打、特性、天气是**三个独立的奠基事件**，应该拆开
- **Gen 4（2006）** — 嘲讽机制+Wi-Fi+VGC起步：✅ 合理，但 VGC 世界锦标赛实际是 2009 年，不是 2006 年——**这是事实性错误**

**问题 A1：事实性错误**
`"title": "嘲讽机制 + Wi-Fi 联网对战 + VGC 起步"` 中的 VGC 起步时间是 2009 年（首届世锦赛），不是 2006 年。

**问题 A2：事件粒度不一致**
奠基类事件中，Gen 3 的"双打+特性+天气"是三件独立大事混为一个事件，而 Gen 5 的"阿罗拉形态"是单一事件。应该统一为**不可分割的最小事件单元**。

#### B. PvP 演进类（pvp）

**问题 B1：禁用信息的来源标注缺失**
所有"被禁用"事件（沙隐、降雨组合、Mega 等）需要明确标注是**官方禁用**还是**社区禁用**。官方禁用的设计权重远高于社区禁用的设计权重。

**问题 B2：未来数据的边界模糊**
2025-2026 的 Pokemon Legends: Z-A 和 Pokemon Champions 是**未来事件**或新发售事件，与 1996-2022 的历史事件性质不同。应该有明确的"当前时间线边界"标注。

#### C. 强化机制演进类（enhancement）

**问题 C1：强化机制的演进脉络不够清晰**
Mega → Z招式 → 极巨化 → 太晶化 是整个报告的核心演进链条，但时间轴没有用任何视觉或结构化手段来呈现这一**纵向演进**。用户看到的只是按时间顺序排列的独立事件，看不到"这四个机制之间的关系是迭代而非孤立"。

#### D. PvE 团体战类（pve）

**问题 D1：贡献归属问题没有与第8条原则建立关联**
时间轴中太晶团体战条目没有引用原则8。**时间轴与原则体系是脱节的**。

#### E. 机制移除类（removed）

这类数据是整个时间轴中最有价值的部分。

**问题 E1：移除机制与相关强化机制的纵向关联缺失**
Mega进化、Z招式、极巨化都是强化机制，它们被移除后又由新机制替代。用户无法在时间轴上看到"Mega进化→Z招式→极巨化→太晶化→Mega进化回归"的完整纵向演进线。

**问题 E2：Pokewalker 归类错误**
Pokewalker 放在 foundation 类，但它的核心设计价值是**"行为绑定奖励"**机制，建议改到 enhancement 或新增一个 meta 类。

#### F. app.py Tab 2 机制时间轴（Plotly 图表）

当前 `app.py:1135-1161` 的 `mechanism_evolution` 数据存在严重问题：

- **问题 F1**：数据完全来自硬编码，没有任何来源标注
- **问题 F2**：Temtem/Cassette Beasts/Palworld 的数据极度贫乏（各仅2条），"多游戏时间轴对照"功能名存实亡
- **问题 F3**：防御机制演进对比表（app.py:1299-1309）是静态硬编码，无置信度标注

#### G. Palworld 时间轴数据

Palworld 时间轴仅有 2 条数据，相比 Pokemon 的 30+ 条，**数据完整度严重不足**。

## 三、交互逻辑审核

### 3.1 当前交互流程

```
用户进入历代时间轴页面
    ↓
选择游戏标签（Pokemon / Palworld / 双轨对照）
    ↓
选择过滤器（全部 / 奠基 / PvP / 强化 / PvE / 已删除）
    ↓
浏览垂直时间轴（按年代倒序排列）
    ↓
点击卡片展开详情
    ↓
查看：detail / 已验证标签 / 移除原因 / 移除分析 / 设计启示 / 影响评估
```

### 3.2 交互缺陷分析

#### 缺陷 1：无法"按设计问题"导航，只能"按时间顺序"浏览

**根本问题**：用户来到时间轴，是想知道"2v2 集火问题是怎么解决的"。但当前的时间轴没有**按问题聚合**的视图。

**改进方向**：提供"按演进链聚合"视图——将"守住→看我嘛→极巨防壁→太晶化"这条线上的所有事件纵向串联。

#### 缺陷 2：展开卡片后的信息层次混乱

当前信息顺序是**从现象到本质**的叙述顺序，但对设计师来说，最有用的信息顺序应该是：

```
设计启示（第一条）
    ↓
移除原因（第二条）
    ↓
影响评估（第三条）
    ↓
详细因果分析（最后）
```

#### 缺陷 3："已验证"标签的语义不清晰

- "已验证"是谁验证的？是 AI？还是人工？验证的标准是什么？
- 一条 detail 有 200 字的信息，如果只有"部分已验证"，用户该信哪部分？

**改进方向**：将"已验证"改为**多级置信度标签**："官方来源" / "社区共识" / "AI 推断·高置信" / "AI 推断·低置信"

#### 缺陷 4："双轨对照"视图的对比价值存疑

当前"双轨对照"只是将 Pokemon 和 Palworld 的事件按年份排序并排显示，但两个游戏在同一年做的事完全没有设计上的对应关系。

**短期改进**：在"双轨对照"视图顶部增加说明。

**长期改进**：重新设计为"设计问题对照"——用户选择一个设计维度，系统展示两个游戏在该维度上的演进差异。

#### 缺陷 5：已删除事件的视觉权重不足

**但"已删除"恰恰是设计师最有价值的参考资料**——它告诉你哪些设计路径是死路。

**改进方向**：
- 增加"只看已删除"的快捷入口
- 将已删除事件的移除原因提取为可搜索标签
- 增加"移除密度图"：按年代显示有多少机制被移除

#### 缺陷 6：缺少与原则/清单的双向链接

时间轴中的一个事件应该**自动关联**到相关原则（原则6：差异化是双刃剑）和相关检查清单条目（A.2.1 强化机制的决策深度）。

#### 缺陷 7：搜索与时间轴的割裂

搜索结果只显示标题和年份，不包含任何上下文。应该增加"关联上下文预览"。

## 四、目的对齐审核

### 4.1 报告的终极目的

> **让游戏设计师能够从 Pokemon 30年的多人对战演进历史中，提取可复用的设计原则和方法论。**

这意味着时间轴的终极价值是**让设计师看到"演进弧线"**——不是一个个孤立事件，而是"问题→解法→后果→下一代解法"的完整链条。

### 4.2 当前满足度评估

| 目的维度 | 当前满足度 | 原因 |
|---------|---------|------|
| 历史事实准确性 | 75% | 有少量事实错误，数据来源标注不完整 |
| 演进脉络呈现 | 40% | 只有横向（时间顺序），缺少纵向（主题演进链）|
| 设计因果揭示 | 60% | removal类事件做得很好，但正常事件缺乏因果分析 |
| 与原则/清单联动 | 20% | 几乎完全断裂，需要人工联想 |
| 多游戏对照价值 | 15% | Palworld数据严重不足，"双轨对照"无语义对齐 |
| 数据可信度透明 | 25% | 只有简单的"已验证"标签，无多级置信度体系 |

## 五、完整改进计划

### 第一阶段：数据质量筑基（P0，需立即处理）

#### P0-1：修复事实性错误

- 将"嘲讽机制+Wi-Fi+VGC起步（2006）"中的 VGC 起步时间修正为 2009（首届世锦赛）
- 区分所有"禁用"事件为"官方禁用"和"社区禁用"两类

#### P0-2：建立多级置信度体系

将当前简单的布尔 `data_verified` 改为枚举 `data_confidence`：

```json
{
  "data_confidence": "official",        // 官方来源（Serebii/Wiki公告）
  "data_confidence": "community",       // 社区共识（Smogon记录）
  "data_confidence": "inferred_high",   // AI推断·高置信（有历史数据支撑）
  "data_confidence": "inferred_medium", // AI推断·中置信（有间接证据）
  "data_confidence": "inferred_low"     // AI推断·低置信（单源推断）
}
```

#### P0-3：统一 app.py Tab 2 与 docs/index.html 的时间轴数据

当前两套数据源独立维护极易产生不一致。应该：
- 将 `app.py:1135-1161` 的硬编码数据迁移到 `report_data.json`
- `app.py` 从 `report_data.json` 读取，或直接调用 `generate_report_data.py` 生成的数据

### 第二阶段：信息结构优化（P1，本季度内）

#### P1-1：事件粒度标准化

- Gen 3 的"双打+特性+天气"拆分为 3 个独立事件
- Gen 4 的"嘲讽+Wi-Fi+VGC起步"拆分为 3 个独立事件（VGC 起步时间修正为 2009）
- 统一每条事件的 detail 长度在 50-200 字之间

#### P1-2：建立演进链纵向索引

为每条事件增加纵向演进链标签：

```json
{
  "evolution_chain": "enhancement",
  "chain_position": 2,
  "chain_prev": "Mega进化",
  "chain_next": "极巨化",
  "chain_related": ["战斗前沿移除", "三打移除"]
}
```

#### P1-3：修正 Pokewalker 分类

将 Pokewalker 从 foundation 改为 enhancement 或新增 meta 类。核心设计贡献是"行为绑定奖励"机制。

#### P1-4：扩充 Palworld 时间轴数据

至少需要扩充到：
- 每次大版本更新的核心机制变化（目前缺失 DLC 内容）
- PvP 竞技内容的演进（目前缺失）
- 服务器/架构变更（目前缺失）

### 第三阶段：交互体验提升（P2，下季度）

#### P2-1：增加演进链聚合视图

在现有的"游戏标签+类型过滤器"之外，增加第四个过滤器维度："演进链"：

```
全部
按游戏：Pokemon / Palworld / 双轨对照
按类型：奠基 / PvP / 强化 / PvE / 已删除 / 社交
按演进链：强化机制链 / 防御机制链 / 团体战链 / Meta管理链
```

#### P2-2：重新设计卡片展开信息顺序

将"已删除"类卡片的信息顺序调整为**学习效率最优顺序**：

```
设计启示（第一条，高亮显示）
    ↓
影响评估标签（positive/negative/mixed/neutral）
    ↓
移除原因（第二条）
    ↓
详细因果分析（收起，需要展开）
    ↓
原始 detail（最后）
```

#### P2-3：增加移除密度热力图

在时间轴顶部增加一个横向热力图，显示每个年代被移除的机制数量：

```
Gen 3 | Gen 4 | Gen 5 | Gen 6 | Gen 7 | Gen 8 | Gen 9
  0      0      1      1      8      2      3
```

Gen 7 的移除高峰（8条）是 Game Freak "简化策略"最激进世代的重要设计信号。

#### P2-4：双向关联系统

每条时间轴事件增加"关联阅读"：

```json
{
  "related_principles": [6],
  "related_checklist": ["A.2.1", "A.2.4"],
  "related_events": ["Mega进化", "Z招式移除"]
}
```

前端渲染时，每条展开卡片底部显示：

```
📚 关联设计原则
  原则6：差异化是双刃剑（谁能用→怎么用）
  原则4：使用决策优于数值

📋 关联检查清单
  A.2.1 强化机制的决策深度检查

🔗 演进链
  ← Mega进化（Gen 6） | 太晶化（Gen 9）→
```

#### P2-5：改进"双轨对照"视图

短期：在视图顶部增加免责声明，说明当前只是年份并置，不是语义对齐。

长期：重新设计为"设计问题对照"——用户选择设计维度（强化/防御/团体战），系统展示两个游戏在该维度上的演进差异分析（由 AI 生成）。

#### P2-6：改进搜索上下文预览

搜索结果中每条时间轴结果增加：

```
[时间轴] 太晶团体战（Gen 9）
  → 关联原则：原则8（可见性优于隐藏性）
  → 关联清单：A.5.5
  → 上下文：贡献归属问题仍未完全解决
```

### 第四阶段：生态联动（P3，持续迭代）

#### P3-1：打通时间轴与 Streamlit app.py Tab 4

让时间轴成为演进报告的**历史验证层**——当 AI 生成一个演进报告时，自动将报告中的结论与时间轴中的历史事实进行交叉验证。

#### P3-2：数据新鲜度追踪

对于新事件（如 Pokemon Champions），需要有数据新鲜度标注：

```
Pokemon Champions 时代开启（2026年4月8日上线）
⚠️ 数据新鲜度：高（近期事件，可能持续更新）
```

#### P3-3：建立时间轴编辑工作流

建立结构化的数据编辑流程，每条事件记录编辑历史和数据来源。

## 七、一句话核心建议

**历代时间轴当前最大的问题不是 UI 丑，而是它扮演了错误的角色——它现在是一个"历史年表"，但它的正确角色应该是"设计演进的证据索引"。**

改进的核心方向：**从"事件列表"升级为"设计演进证据系统"**——每条事件都应该能回答"这个设计为什么出现在这个时间点""它解决了什么问题""产生了什么后果""与哪些其他事件有因果关联"。
