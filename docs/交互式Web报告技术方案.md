# 交互式 Web 报告技术方案

> 目标：将综合研究报告（`综合研究报告_宝可梦VGC多人对战设计经验.md`）重构为可交互的 Web 应用。

---

## 核心交互设计

```
┌─────────────────────────────────────────────────────────────┐
│  [Header: 报告标题 + 筛选控件]                              │
├──────────────┬──────────────────────────────────────────────┤
│              │                                              │
│  左侧导航    │  右侧内容区                                   │
│  (锚点导航)  │                                              │
│              │  ┌────────────────────────────────────────┐  │
│  · 10条原则  │  │  原则卡片网格（可展开详情）              │  │
│  · 分章节    │  └────────────────────────────────────────┘  │
│  · 检查清单  │                                              │
│  · 对照视图  │  ┌────────────────────────────────────────┐  │
│  · 附录      │  │  可交互时间轴（历代演进）                │  │
│              │  └────────────────────────────────────────┘  │
│              │                                              │
│              │  ┌────────────────────────────────────────┐  │
│              │  │  检查清单（可勾选 + 优先级筛选）          │  │
│              │  └────────────────────────────────────────┘  │
│              │                                              │
└──────────────┴──────────────────────────────────────────────┘
```

### 1. 左侧锚点导航

- 固定在左侧，可折叠
- 包含：10条原则速查 → 分章节 → 检查清单 → Pokemon/Palworld 对照视图 → 附录

### 2. 原则卡片网格（首页）

- 10条原则以卡片形式展示（3列网格）
- 每张卡片：编号 + 原则名称 + 一句话总结
- 点击展开：历史案例引用 + 对应检查清单条目
- 颜色编码：平衡性（蓝）/ 强化机制（橙）/ 多人玩法（绿）/ Meta管理（紫）

### 3. 检查清单交互

- 可勾选状态 + 勾选进度百分比
- 优先级筛选：⚠️核心 / 重要 / 一般
- 按维度筛选：PvP / PvE / Meta管理 / 养成
- 已勾选项高亮相关设计原则

### 4. 历代演进时间轴

- Pokemon：1996-2026（29年）
- Palworld：2024-2026（2年）
- 双轨时间轴，点击展开对应版本的详细更新内容
- 支持按游戏筛选

### 5. Pokemon vs Palworld 对照视图

- 双栏并列对照（维度选择下拉框）
- 维度：PvP模式 / 团体战 / 强化机制 / 平衡管理 / 多人系统

### 6. 可折叠延伸内容

- 特定术语 hover 显示历代演进 tooltip
- 历史数据展开区块（按需加载）

## 技术选型

### 方案 A（推荐）：纯 HTML/CSS/JS 单文件

**优势**：
- 无需构建工具，直接浏览器打开使用
- 部署最简单：放到任意 Web 服务器或 GitHub Pages
- 无依赖，安全（无 npm供应链风险）
- 离线可用（单文件）
- 极低维护成本

**劣势**：
- 大型组件复用不如框架方便（但报告规模用 vanilla JS 完全够用）
- 不适合深度交互（但报告主要是阅读+勾选+跳转，不需要复杂状态）

**适合场景**：作为工具使用，不作为持续开发的产品

### 方案 B：Vue 3 + Vite

**优势**：
- 组件化架构，适合长期维护
- 生态丰富（路由、状态管理）
- 社区活跃

**劣势**：
- 需要 Node.js 环境构建
- 增加部署复杂度
- 对于"阅读+勾选"类场景，过于重量

**适合场景**：如果报告需要持续迭代、添加更多游戏数据

### 方案 C：Streamlit

**优势**：
- 复用项目已有技术栈（Python）
- 快速出原型
- 内置数据可视化

**劣势**：
- UI 不够精细（偏向数据 app 而非阅读报告）
- 部署需要 Python 环境
- 交互体验不如 SPA

**适合场景**：快速出 demo，不适合正式发布

### 方案 D：Next.js + Tailwind CSS

**优势**：
- 现代化的开发体验
- 静态生成（SSG），SEO 友好
- Tailwind 快速美化

**劣势**：
- 需要 Node.js 构建
- 配置复杂度最高
- 对于纯报告场景过于复杂

**适合场景**：如果未来要扩展为多语言/多游戏数据门户

## 推荐方案：方案 A（纯 HTML）

**理由**：
1. 本项目已有 Python 技术栈，报告生成可以考虑用脚本从 Markdown 提取数据
2. 报告本质是"阅读工具"，不需要复杂的 SPA 状态管理
3. 单文件 HTML 可直接嵌入到 app.py 作为子页面
4. 可以先做 HTML 版本验证交互，之后按需迁移到 Vue/React

### 实现路径

```
Phase 1: 单文件 HTML 交互报告
  - 从 Markdown 自动提取数据（Python 脚本）
  - 生成包含 JSON 数据 + 交互逻辑的单 HTML 文件
  - 验证交互设计

Phase 2: 扩展（如有需要）
  - 添加 Pokemon vs Palworld 时间轴可视化
  - 添加多语言支持
  - 迁移到 Vue（如需持续迭代）
```

## 数据提取方案

报告中的数据（10条原则、检查清单、历代演进时间轴）可以从 Markdown 提取为 JSON，供前端使用。

Python 脚本 `scripts/generate_interactive_report.py`：
1. 解析 `综合研究报告_宝可梦VGC多人对战设计经验.md`
2. 提取 10 条原则、检查清单、章节结构
3. 合并 Palworld 对照数据
4. 输出 `docs/report_data.json`

HTML 文件 `docs/index.html`：
1. 内嵌 `report_data.json`
2. 实现所有交互逻辑
3. 响应式设计（支持移动端）
