# 设计演化档案 — 工具设计审核报告

> 审核日期：2026-04-17
> 审核模式：多角色交叉审核
> 审核工具版本：1.7.1

---

## 一、审核小组介绍

本审核小组由 5 个角色组成，每个角色代表一类关键利益相关方，通过独立审视、交叉辩论的方式对工具进行全面审核。

| 角色 | 定位 | 核心关注点 |
|------|------|-----------|
| **制作人** | 产品最终负责人 | 产品定位、用户价值、迭代优先级、ROI |
| **营销视角** | 市场与用户增长 | 受众定义、工具卖点、传播路径、品牌一致性 |
| **技术负责人** | 工程实现保障 | 架构合理性、扩展性、技术债务、可维护性 |
| **UX 设计师** | 用户体验守护者 | 交互流程、信息层次、认知负荷、无障碍性 |
| **数据分析师** | 数据质量守门人 | 数据完整性、分析结论可靠性、偏差与幻觉风险 |

## 二、工具设计目的复盘

### 2.1 核心价值主张

> 通过真实更新日志数据 + LLM 分析，追踪和理解多人对战游戏的设计演进逻辑。

工具声称解决的核心问题：

1. **信息分散**：宝可梦Like游戏的更新日志散布在多个平台（Wiki、Steam News、Bulbapedia），没有统一入口
2. **意图不透明**：玩家社区对设计改动的讨论往往流于表面，缺乏系统性分析
3. **研究成本高**：理解一个设计主题（如"集火与保排"）需要跨越多个世代、多个信息源手动整理

### 2.2 架构总览

```
用户交互层（Streamlit app.py）
    │
    ├── Tab1 版本编年史
    ├── Tab2 机制时间轴（Plotly）
    ├── Tab3 设计意图分析（LLM 单条）
    ├── Tab4 演进报告（AI主题发现 → 分层主题树 → 报告生成）
    └── Tab5 版本对比

数据管理层（DataManager）
    ├── Steam游戏 → 本地 JSON（Temtem/Palworld/Cassette Beasts）
    ├── Pokemon → 内置结构化数据（scrapers/pokemon_wiki.py）
    └── Pokemon VGC历史 → data/pokemon/vgc_history.json

AI 分析层
    ├── IntentExtractor     — 设计意图提取（LLM）
    ├── AITopicDiscoverer   — AI 动态主题发现
    ├── ReportGenerator     — 演进报告生成
    └── Prompts            — Prompt 模板库

数据持久化层
    ├── SQLiteStore — 结构化数据
    └── VectorStore — ChromaDB 向量检索
```

### 2.3 工具当前支持的 5 款游戏

| 游戏 | 数据来源 | 多人对战特征 |
|------|----------|-------------|
| Pokemon 剑/盾 | 内置数据 + Wiki | VGC 双打 |
| Pokemon 朱/紫 | 内置数据 + Wiki | VGC 双打 + 团体战 |
| Temtem | Steam News API | 2v2 战斗 |
| Cassette Beasts | Steam（无公告） | 合作战斗 |
| Palworld | Steam News API | PvP/PvE |

## 三、各角色独立审视报告

### 3.1 制作人视角

**核心问题 #P1：演进报告生成速度与用户期望严重不匹配**

- 报告 UI 显示"预计需等待 30-120 秒"，但没有进度反馈机制
- 用户在 Streamlit 的等待体验几乎是黑屏，没有任何中间状态
- 如果报告生成失败（LLM 超时或格式错误），用户不知道该重试还是换主题
- **影响**：用户流失率会非常高，尤其在第一次体验时

**核心问题 #P2：演进报告新流程（分层主题树）存在明显的状态管理混乱**

- 状态变量过多：`sel_dimension_idx`、`sel_branch_idx`、`selected_topic_idx`、`sel_dim`、`sel_branch` 等至少 7 个相关状态
- 存在"旧版兼容分支"（`if use_topic_tree else`），意味着两个 UI 路径同时维护
- `report_key` 的生成逻辑复杂且分散在 UI 逻辑中，容易出现缓存不刷新的 bug
- **影响**：长期维护成本高，bug 排查困难

**核心问题 #P3：工具定位模糊——面向谁？是工具还是内容？**

- 如果是**研究工具**，应该有更好的导出、引用、API 接口
- 如果是**内容消费**，应该有更好的阅读体验（如 Markdown 渲染美化、目录跳转）
- 当前两者都做了一点，但都不够好
- **营销视角的挑战**（见 4.2）："这个工具的 MVP 用户是谁？他们的付费意愿是什么？"

**制作人对数据分析师的挑战**：工具依赖 LLM 实时生成分析，但 LLM 的输出是不确定的。如果两份同样输入生成了内容差异巨大的报告，用户会不会认为工具"不可靠"？这会不会毁了工具的口碑？

**制作人对 UX 的挑战**：Tab 4 演进报告现在的信息密度极高——主题树 + 演进阶段 + 报告内容堆在一起，普通用户真的能理解这套层层嵌套的交互吗？

### 3.2 营销视角

**核心问题 #M1：工具缺乏清晰的"杀手级场景"**

- 当前 README 描述了 6 个功能点，但没有说清楚"用户在哪一刻会打开这个工具"
- 竞品对比缺失：Wiki、Reddit、Serebii 都能查到更新日志，这个工具提供的增量价值在哪里？
- 6 个功能中，"语义检索"（用自然语言问问题获取历史设计知识）是最具差异化的，但却是展示最弱的功能
- **结论**：工具没有明确的"第一印象锚点"

**核心问题 #M2：受众定义不清**

- "宝可梦Like游戏多人对战设计演进研究"这个定位非常小众
- 小众用户有更强的传播能力（核心玩家社区），但绝对数量有限
- 营销素材（如 README 的 README 的设计意图分析部分）写得像学术论文，不吸引普通玩家
- **风险**：可能永远停留在"个人项目"级别

**核心问题 #M3：品牌感知不统一**

- 工具名称是"设计演化档案"，但内部大量文件注释、Prompt 模板仍然包含"宝可梦"特定术语（虽然刚刚修改过 prompts.py，但整个数据层和分析层仍有大量残留）
- README 中"支持的游戏"表格里 Pokemon 相关内容占比 40%，但工具声称是"通用研究工具"
- 用户进入后感知到的"专业化程度"与宣传文案不完全匹配

**营销对制作人的挑战**：如果这个工具的第一批用户就是你们自己（研究宝可梦设计演进），那它本质上是一个**内部研究流程**，不是产品。这种情况下，营销视角的审核是否还有意义？还是说我们应该先回答"这个工具要不要对外发布"这个问题？

### 3.3 技术负责人视角

**核心问题 #T1：Prompts 模块存在未完成迁移**

- `analyzer/prompts.py` 中包含三个 Prompt：
  - `DESIGN_INTENT_PROMPT` — 引用 `{GAME_DESIGN_CONTEXT}`（已通用化）
  - `SIMPLE_EXTRACT_PROMPT` — **仍然包含** `"pokemon_mentioned": ["提到的宝可梦名称"]`（第 141 行）
  - `TIMELINE_PROMPT` — 已通用化
- `SIMPLE_EXTRACT_PROMPT` 被 `IntentExtractor.classify_patch()` 调用，这个方法可能在多个场景使用
- **影响**：这个 Prompt 的输出 schema 会包含 Pokemon 特定字段，当处理非 Pokemon 游戏时会产生误导性的空字段

**核心问题 #T2：主题发现流程存在架构断裂**

- `TopicDiscovery` 类（第 10-17 行）依赖 `RESEARCH_TOPICS`（空列表）和 `filter_patches_for_topic()`（返回 `patches`），这是旧版兼容代码
- 新的主题发现流程依赖 `AITopicDiscoverer`（AI 动态发现），但 `TopicDiscovery` 类完全没有用到 AI
- 两条路径并存：`topic_discovery.py`（旧，基于关键词）和 `ai_topic_discoverer.py`（新，基于 LLM）
- `report_generator.py` 使用 `AITopicDiscoverer`，但 `topic_discovery.py` 和 `research_topics.py` 的兼容代码仍然存在且被 app.py 引用
- **影响**：维护两套主题发现逻辑增加了认知负担，新工程师可能困惑该走哪条路径

**核心问题 #T3：向量检索与结构化存储的职责边界不清**

- `IntentExtractor.get_vector_store()` 每次调用都会重新初始化 `VectorStore` 实例，没有单例或连接池管理
- `VectorStore` 的 schema 设计不明确——存入的是哪类文本（原始更新日志？分析后的 JSON？还是两都有）？
- `semantic_search()` 返回的格式不明确，用户得到的是匹配片段还是整条记录？
- ChromaDB 是持久化向量数据库，但代码中没有看到向量数据的导出/备份逻辑

**核心问题 #T4：错误处理过于简单**

- `IntentExtractor._fallback_analysis()` 是一个简单的关键词匹配降级，但降级输出包含了 `design_pattern`、`risk_assessment`、`related_changes` 等在正常流程中不存在的字段
- `batch_analyze()` 没有并发机制，100 条更新 × 平均 10 秒 = 1000 秒的串行等待
- `report_generator.py` 的异常处理包裹了整个生成流程，一旦任何步骤失败，用户只收到通用错误信息

**技术对数据分析师的挑战**：向量检索的语义相似度阈值是多少？如果用户问"速度线如何影响双打"，返回的前 5 条结果真的相关吗？有没有做过召回率和精确率的评估？

**技术对 UX 的挑战**：Tab 5 版本对比允许用户选择任意两个版本/游戏，但底层有没有做差异对齐（比如 Pokemon Gen 8 的 "Ver 2.0" 和 Temtem 的某个版本如何对齐）？如果没有，用户做的对比可能是在比较两个完全不相关的东西，UX 却给了一个看起来很专业的界面。

### 3.4 UX 设计师视角

**核心问题 #U1：Tab 4 演进报告的信息密度超出了用户的处理能力**

- 当前 UI 同时展示：维度选择 → 演进枝选择 → 演进阶段预览（可折叠）→ 完整报告文本
- 演进时间线预览在展开状态下可能列出 10+ 条补丁标题，视觉上非常拥挤
- 用户需要在 3 层嵌套结构中做选择，但没有清晰的"当前在哪里"的导航
- **认知负荷评估**：用户需要同时理解"维度→演进枝→阶段→报告"这 4 个概念才能有效使用，而普通用户可能只理解了"选一个主题看报告"

**核心问题 #U2：缺失的关键 UX 元素**

- **没有进度条**：LLM 调用期间，Streamlit 的 `spinner` 是在服务端等待，前端用户看到的是不确定的等待
- **没有空状态引导**：如果某个游戏没有多人对战相关的更新，用户只会看到空白页面，不知道是数据缺失还是筛选条件太严格
- **没有撤销/返回机制**：演进枝选错后需要重新选择，但没有清晰的"上一步"操作
- **没有分析结果的可复用性**：生成的报告只存在于 session 内存中，刷新页面即丢失；没有复制、导出、收藏功能

**核心问题 #U3：Tab 3 设计意图分析的输出可读性差**

- LLM 返回的 JSON 直接渲染为原始文本，没有格式化处理
- `design_rationale` 字段可能非常长，但没有分段或高亮关键信息
- 关键字段如 `balance_assessment`（"过犹不及/恰到好处/力度不足/治标不治本"）与普通文本混在一起，用户需要逐字阅读才能理解

**核心问题 #U4：多游戏选择与数据一致性**

- Tab 1/3/4 都有游戏选择，但切换游戏时：
  - Tab 4 的演进报告状态是否正确重置？
  - 如果游戏 A 没有主题数据，Tab 4 应该怎么引导用户？
- 游戏选择器没有显示每个游戏的数据量（如"朱/紫 127条"），用户不知道切换后会有什么

**UX 对技术的挑战**：Streamlit 的 session state 管理演进报告的多步骤选择流程，在高并发（如多人同时使用部署的版本）下，session state 是基于 cookie 的，会不会出现状态混淆？更重要的是，当前没有看到任何状态隔离的测试。

**UX 对营销的挑战**：工具的主色调、Logo、宣传语是什么？如果用户把工具分享到 Reddit/Discord，视觉上有没有辨识度？当前 Streamlit 默认主题是否传递了"专业游戏设计研究工具"的感觉？

### 3.5 数据分析师视角

**核心问题 #D1：LLM 分析结果缺乏可验证性**

- `IntentExtractor.analyze_intent()` 返回的 JSON 字段（如 `design_rationale`、`balance_assessment`）完全由 LLM 生成，没有任何交叉验证机制
- LLM 可能会"幻觉"出不存在的历史案例（如引用一个"不存在的 Gen 6 调整"），且这类幻觉无法被用户识别
- `similar_historical_cases` 字段尤其危险：如果 LLM 编造了一个历史上从未发生的改动，用户可能信以为真
- **风险**：工具的分析结论可能被引用在社区讨论中，而其真实性完全无法保证

**核心问题 #D2：数据来源的权威性不透明**

- Pokemon Gen 8/9 数据标注来自 Serebii.net，但：
  - Serebii 是粉丝维护的网站，存在数据错误和延迟
  - 内置的 Pokemon 数据版本号与实际游戏版本是否对齐？
  - `scrapers/pokemon_wiki.py` 内置数据是什么时间点采集的？是否有版本追踪？
- Steam News API 数据（Temtem/Palworld）：
  - Steam News 的更新日志是否完整？某些游戏不在 Steam 上发布所有更新
  - 没有数据新鲜度提示，用户不知道游戏数据是 1 周前还是 1 年前采集的
- **关键缺失**：没有任何"数据质量报告"——如每款游戏有多少条更新，其中有多少条与多人对战相关

**核心问题 #D3：分类和筛选的准确性未评估**

- `DataManager._categorize()` 使用硬编码关键词匹配来分类更新
- 关键词列表（如 `CATEGORY_RULES["PvP"]`）包含英文 + 中文，但没有大小写标准化（代码中用了 `.lower()` 但关键词本身有大小写混用，如 `"VGC"` 和 `"vgc"`）
- "多人对战"的过滤逻辑只基于关键词，不验证语义相关
- **实测风险**：一条关于"游戏打折促销"的更新如果恰好包含 "battle pass" 关键词，会被误分类为 PvP

**核心问题 #D4：主题发现的代表性偏差**

- `AITopicDiscoverer` 依赖 LLM 自动发现主题，但 LLM 发现的主题受其训练数据影响，可能偏向某些社区讨论热度高的主题
- 如果某个游戏的设计演进在社区讨论少，AI 可能完全忽略该主题
- 当前没有任何机制让用户知道"这个主题在数据中的代表性如何"（如 5/100 条更新 vs 50/100 条更新，代表的权重完全不同）
- `TopicDiscovery.match_ratio` 字段计算的是"匹配数/总更新数"，但这个比例本身没有归一化——主题 A 有 20 条更新但只占总数据 5%，主题 B 有 5 条更新但占总数据 50%，代表意义完全不同，但 ratio 值会掩盖这一点

**数据分析师对技术的挑战**：`SIMPLE_EXTRACT_PROMPT` 输出 `pokemon_mentioned` 字段，但工具声称是通用研究工具。如果处理 Temtem（一个没有"宝可梦"概念的物种系统），这个字段会返回什么？如果是空列表，用户会不会误以为 Temtem 没有生物单位？

**数据分析师对 UX 的挑战**：当用户看到 `balance_assessment: "过犹不及"` 时，他们真的能理解这个词在游戏设计语境下的含义吗？普通用户会不会把这个评估误解为"这次改动很糟糕"？有没有考虑过用更直白的语言重新表达？

## 四、第二轮：交叉辩论记录

### 辩论 1：制作人 vs. 数据分析师
**主题：LLM 输出不确定性与工具可靠性**

> **制作人**：如果两份同样输入生成了内容差异巨大的报告，用户会不会认为工具"不可靠"？

**数据分析师回应**：这是 LLM 研究工具的固有特性，无法完全消除，但可以通过以下方式缓解：
1. 每次分析记录输入的 temperature、model version、prompt version，确保可复现
2. 在报告中显示"分析置信度"（如基于多少条相关历史案例）
3. 对高风险字段（`similar_historical_cases`）增加事实核查提示
4. 核心结论需要用户主动确认，而不是被动接受

> **制作人追问**：但这会增加开发成本，而且可能会让界面变得更复杂。你们建议在什么阶段做这件事？

**数据分析师**：优先级建议：**立刻做** `similar_historical_cases` 的标注——因为这是幻觉风险最高的字段，同时对用户的误导性最强。其他字段可以放在 v2。

---

### 辩论 2：营销 vs. 制作人
**主题：工具的 MVP 定义**

> **营销**：如果这个工具的第一批用户就是你们自己，那它本质上是一个内部研究流程，不是产品。营销视角的审核是否还有意义？

**制作人回应**：这个问题问到了核心。当前工具确实处于"个人项目"阶段，但即使是个人项目，也有两种路线：
- **路线 A**：继续做深 Pokemon 社区的专业研究工具 → 用户是硬核 VGC 玩家和研究人员 → 有明确的社区传播路径
- **路线 B**：扩展为通用游戏设计研究工具 → 竞争对手变成 Notion + AI 研究助手 → 需要更多差异化

> **营销追问**：那当前 README 的受众是谁？如果是 VGC 玩家，README 中"集火与保排"、"爆发资源"这些术语对他们太陌生；如果是研究者，"快来试试"的口吻又太随意。

**制作人承认**：当前 README 是写给自己的，没有认真考虑过外部受众。**这是一个 P1 问题**。

---

### 辩论 3：技术 vs. UX
**主题：版本对比的数据对齐问题**

> **技术**：Tab 5 版本对比允许用户选择任意两个版本/游戏，但底层没有做差异对齐。Pokemon Gen 8 的 "Ver 2.0" 和 Temtem 的某个版本如何对齐？用户做的对比可能是在比较两个完全不相关的东西，UX 却给了一个看起来很专业的界面。

**UX 回应**：完全同意。这涉及到信息架构的根本问题：
1. **"版本"这个概念在不同游戏中有完全不同的含义**：Pokemon 是"世代+版本+补丁"，Temtem 是"赛季"，Palworld 是"大版本"
2. 在没有语义理解的情况下，"比较版本 A 和版本 B"本质上是一个**跨游戏设计比较**问题，需要 LLM 来做语义对齐
3. 当前的版本对比功能实际上是"把两条更新日志并排放在一起"，**没有任何分析价值**，反而可能误导用户

> **技术追问**：那 UX 的建议是什么？

**UX 建议**：
- **选项 1（保守）**：给版本对比增加提示："此功能仅展示原始更新内容，不做语义对齐，分析价值有限"
- **选项 2（激进）**：版本对比改为"设计主题对比"——让用户选择两个主题/演进枝，让 LLM 对比这两个主题在不同游戏中的演进差异
- 技术评估：**选项 2 需要重构底层数据模型，当前版本无法实现**

---

### 辩论 4：各角色 vs. 演进报告新流程
**主题：分层主题树的价值与成本**

> **集体质疑**：演进报告的分层主题树设计是否过于复杂？

**制作人**：新流程的目的是让用户更好地选择研究主题——不是让用户在 50 个扁平主题里大海捞针，而是通过维度→演进枝的层次结构引导。但代价是状态管理复杂度急剧上升。

**UX**：引导逻辑是对的，但**视觉语言不够清晰**。维度用数字编号（0、1、2...），演进枝用字母编号（A、B、C...），但没有任何图例说明这是什么含义。首次用户打开主题树，不知道该从哪里开始。

**技术**：同意。更严重的是，`ai_topic_discoverer.py` 生成的主题树是动态的（每次调用 LLM 可能不同），这意味着用户在不同 session 可能看到不同的主题树结构。**工具缺少"主题树的稳定性机制"**——应该缓存生成的主题树，而不是每次重新生成。

**数据分析师**：主题树中每个节点应该显示**数据置信度**。如果某个演进枝只有 2 条相关更新，AI 生成的分析结论可靠性远低于有 20 条更新的演进枝，但用户看到的是完全相同的 UI。

## 五、集体讨论结论

经过两轮交叉辩论，审核小组就以下核心问题达成共识：

### 结论 1：工具当前处于"功能完整但体验破碎"阶段

工具的核心 AI 分析能力（IntentExtractor、演进报告生成）已经具有真实价值，但：
- 用户界面没有充分体现这个价值（信息密度过高、可读性差）
- 缺少关键体验元素（进度反馈、导出、状态引导）
- 部分功能（版本对比）的分析价值存疑

### 结论 2：演进报告是工具的核心差异化功能

Tab 4 演进报告是工具唯一不可替代的功能（结合真实数据 + LLM 生成专题分析），应该获得最高的迭代优先级。所有其他 Tab（编年史、时间轴、版本对比）都可以用 Wiki + 简单脚本来替代。

### 结论 3：数据质量问题是最大的系统性风险

如果工具的分析结论被社区引用，但数据来源不透明、LLM 幻觉未标注、分类准确率未验证，长期会损害工具的可信度。这个问题不会立刻显现，但会在工具获得第一批真实用户后逐渐暴露。

### 结论 4：Prompts 模块需要一次完整的清理

`prompts.py` 的通用化改造不彻底，`SIMPLE_EXTRACT_PROMPT` 仍包含 Pokemon 特定字段；`research_topics.py` 的兼容代码是技术债务；两条主题发现路径并存增加了维护复杂度。

## 八、审核总结

设计演化档案的核心 AI 分析能力具有真实的用户价值——将零散的更新日志转化为有结构的设计演进知识，这一工作本身是有意义的。但工具目前处于"技术可行性验证阶段"向"产品化阶段"过渡的关键节点，存在三个主要的成熟度问题：

1. **体验成熟度不足**：核心功能的交互体验（进度、错误、导出）远未达到用户期望
2. **数据可信度不透明**：LLM 分析结论缺乏事实核查，数据来源缺乏权威背书
3. **产品定位不清晰**：工具在"研究工具"和"内容平台"之间的边界模糊

这三个问题的优先级为：**体验问题（P0）** > **可信度问题（P1）** > **定位问题（P1/P2）**。

建议下一版本的迭代重点放在 **P0 问题的修复**（Prompts 清理 + 进度反馈 + 错误处理），在此基础上再逐步解决 P1 问题，最终在 P2 阶段完善产品定位和品牌建设。
